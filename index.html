    What is machine learning

    In the modern day, machine learning is the invisible technology that surrounds us. It is always there behind the scenes processing data. But what is it exactly? Essentially machine learning is a process in which a computer is fed a series of instructions and then set to work to identify and categorize data. An example of machine learning we see in our everyday lives is on Spotify. When we listen to a song on Spotify, the application will take that information and then supply the user with other songs that are similar. We, the user, give the machine information about what we enjoy, and it is then able to process it and suggest other songs for us to listen to. There are three main machine learning algorithms that see wide use in the present, with a fourth gaining popularity. Supervised Machine Learning Unsupervised Machine Learning Reinforcement Learning These each have their own benefits and disadvantages as well as different uses in a variety of contexts but will continue to see usage for years to come. Supervised Machine Learning is an algorithm where we input information that the program can then use to identify and categorize future inputs. For example, if we were to feed the program hundreds of images of cats and dogs, with each image being labelled, the algorithm would ideally be able to understand aspects of what makes a cat a cat, versus a dog. It will start looking for several aspects, like the shape of the eyes, ears, body and tail, the algorithm will then categorize it as either a cat or a dog. We commonly see this algorithm utilized in fraudulent email protection, determining a loan applicant‚Äôs viability and detecting fraudulent bank withdrawals. Unsupervised Machine Learning is a somewhat messier approach. Essentially instead of teaching the algorithm what the information we are putting into it is, we will instead just allow the program to arbitrarily sort through the information and decide how to categorize it. This is useful to identify otherwise hidden patterns in datasets. This is widely seen in identifying customer‚Äôs behaviour, grouping it with other customers and recommending products based on the buying behaviour of other customers in the same group. Reinforcement Learning is most closely related to how humans learn. Instead of being fed pre-labelled datasets, a machine running the algorithm is instead allowed to interact with its environment. The machine is then rewarded for its good behaviour and punished for its bad behaviour. Over time it is expected the machine can determine what it should do as opposed to what it shouldn‚Äôt do. This algorithm is often seen in robotics, self-driving cars and AI designed to play games. Finally, a fourth algorithm is not fully embraced by everyone but is set to see more use in coming years. Semi-supervised Learning is a hybrid of supervised learning and unsupervised learning, hence the name. The algorithm is fed a small portion of labelled data, as well as a larger amount of unlabelled data, using this data it can sort the data to a high degree. Semi-supervised learning is still considered to be relatively new compared to the other methods but has already proven effective to minimize the issues that arise when using the other methods. These algorithms are all seeing constant use presently but there are new innovations on the horizon, as with most cutting-edge technologies. Automated Machine Learning is already seeing some use but is expected to become much more refined in the near future.

    Project Ideas

    Overview: Goblin

    Goblin will be a python application aimed at providing a way of recording the chat histories of commonly used communication services like slack, discord, and the RMIT forums.

    Unlike chat aggregators for managing chat interactions across multiple platforms that work by sending a copy of the message to a central location. Goblin will be an application that text is pasted into, allowing for ‚Äúretro-active‚Äù recording of text history.

    The point of Goblin is to allow the user to manage small to medium sized chat histories so that they can be organized, searched, and posted on different services.

    Goblin will be an application with some minimal UI a text box, the Idea is that the user selects all the relevant channel history and copy pastes it into the text box (hence the name, Goblin gobbles the text you put IN). The application then separates the wall of text into individual messages and records the name of the user who posted the message, the time stamp on the message, and the content of the message. Those messages will form a ‚Äúsnapshot‚Äù which can then be saved to an existing history or saved as a new history.

    The aim is to make it easy to store all of one‚Äôs chat histories so that they can be searched on and organized, so that it can be easily moved to different platforms or simply saved for later.

    Context and goals

    Why?


    Currently it‚Äôs common to use a wide array of communication methods regularly. For example, right now as I type this, I have discord, slack, MS Teams, Gmail, and Outlook open on my PC. Now, in our daily life we are unlikely to find a need to record or move what goes on in one of those services to a different one, and on the off chance that you do, copy and pasting or screenshotting is usually good enough. However, in a hypothetical scenario where you are a student at RMIT and you are required to use the canvas forums, and later MS Teams, you may find yourself wishing that what was on the forums or the unofficial discord was also on teams‚Ä¶. Enter Goblin. Goblin is aimed at making it easy to record and repost chat histories from different services

    How will it work?

    The application can be thought of in 4 main parts

    Input: The input is the text that gets pasted in. this will then be sent to the middleware module to be processed

    middleware: Once something has been entered, it needs to be broken up and stored. it‚Äôs expected that large amounts of text will be pasted in at once, so the program needs to break that text up into the original messages and store them. Data editing and deletion should be supported as well. When histories are stored there should be an option to add descriptions and tags, summaries as metadata.

    Output: When the database is searched it will output the desired chat.

    The UI: A user interface that combines the input and output features to make creation, search and deletion of chats easy and intuitive.

    The aim make it easy to store all of ones chat histories so that they can be searched on and organized, all in one place

    Use-case

  • A searchable archive of all your chat histories
  • if you want to find your response to a question on a forum that you no longer have access to, you can find it in your Goblin records.
  • Making copying histories from one place to another while retaining a readable format easy.
  • If you had a really informative private conversation about a project on discord, and would like to make it available to the group on MS Teams, Goblin could be used to format and post that entire conversation to teams.
  • Reformatting text chat ‚Äúdumps‚Äù
  • Because Goblin reformats the pasted chat dump into a standard format, so that it can filter messages by date, sender, or keywords, it can provide data for analysis of chat records where the data is kept in its original form.
  • Understanding data from unintegrated sources
  • For example, if you wanted to know how many times the phrase ‚ÄúI hate MS Teams‚Äù was used on the unofficial Intro to Computer Science discord. Once you had stored the data in Goblin you would be able to know both the answer to that question and be able to see what messages contained the phrase. What‚Äôs unique about Goblin is that collecting this data doesn‚Äôt require an integration with the service that you‚Äôre taking the data from. No API calls, nothing.
  • The ability to migrate histories of entire platforms from one place to another

    • My personal project from ITT assignment 1 was a concept like Goblin, the idea was to forward the canvas forum update emails to a slack bot that would post the contents of the forum posts to a slack server, essentially automatically replicating the RMIT forums on slack. I received feedback that it was an interesting concept and could have potential use as an internal tool for a team testing the viability of moving the forums to slack. Goblin aims to be a more robust tool designed to assist in that sort of endeavor. It isn‚Äôt as sleek and fancy as something that is fully automatic and integrated, but it can be applied to a wide array of platforms with no additional programing.

    Technology:

    The key technologies that need to function for the application to accomplish what it should are as follows: Record management functionality: The app will need to do things like; save to an existing history, save as new history, delete history. As well as some filtering functionality that allows users to sort/locate individual messages or display sections of a history. This will be handled by SQLite a free SQL database manager that is easy to implement and widely used. SQLite was chosen as opposed to a larger package like mySQL because its lighter weight and simpler to use for a python program like Goblin. SQLite is also appealing because it runs on the local machine, instead of being cloud based like some other options I looked at. SQLite is also a relational database manager. A relational database is ideal for this use case because of how they‚Äôre intrinsically designed to associate one set of information (in this case date, source, subject matter, original sender as well as any other information that was attached when saved) with another (the content of the message). SQLite is a very commonly used tool but it was chosen mainly for how easy it is to implement, and use was the main reason I think it should be used for the database management.

    Managing the data:

    The point of Goblin is primarily a data manager for message histories. So, if nothing else the data management should well implemented. SQLite will be what we use to handle this, and it was chosen specifically because it‚Äôs lightweight and easy to use.

    Having not been trained in SQLite at all, I spent about two or three hours talking to my dad about what needs to be done there to achieve what goblin needs. This is my preliminary understanding; there needs to be at least 2 tables, one that contains information attached to the actual snippet (an individual message) and another table for the tags that the user may specify about a given chat dump.

    The tag table needs to be separate because if it wasn‚Äôt it would massively complicate editing the tags if they were tangled up in the rest of the data about a snippet. The tags are the main way users can define information about what they‚Äôre storing. It‚Äôs expected that even with tags, what‚Äôs stored in goblin could get very messy, at this point in the design its up to the user to keep everything readable using the tagging method, so editing, deleting, and adding tags is a very important feature.

    The snippet table contains the inferred information like the author, date, and tag ID. As well as a user defined field called source, where the user defines where the snippet came from. The table also contains the snippet itself. The tag ID is what associates the tags in the tag table with the snippet table

    A bare bones UI The UI will be created using tkinter in python. It will need to have buttons, output tables and input boxes for the basic functionality of Goblin.

    In general, I propose to model the UI after the slack or discord UI except instead of channels on the left bar there‚Äôs the metadata tag list, and where the messages would be in slack is where the selected file displayed. Text formatting capability: The ways different services format their content need to be accounted for, ideally Goblin should be able to store most common forms of written communication emails, slack messages, discord messages and forum posts etc. Goblin will need to have a way of transforming the content of all these formats so that it can filter the information properly. This will be developed by the group in python.

    Challenges & required skills:

    Sorting

    Sorting the messages, isolating the sender, the time, and the message content is tricky given that there is only loose consistency in the formats we can expect. This will require a backend solution to be developed, something that ‚Äúrecognizes‚Äù the content based on the source tag. Or a front-end UI based solution where the user tells the program how the text is formatted e.g.; this string is who sent the message, this is the time stamp, and if all the other messages are formatted the same way.

    Building the UI

    This is something that I don‚Äôt think I, or anyone in my group has experience with, although the requirements are quite common things to do in python/windows. Simply because we‚Äôve never done anything like this there may be planned features that turn out to be quite difficult to implement.

    Handling un-transportable content:

    Some content is not going to be accessible when copied, such as some emoji‚Äôs, links to sites, images etc. these will show up as URLs in the input box. I‚Äôm not sure how they should be handled from a UX point of view. Removing them, if possible, may make messages less readable. Leaving them in my mean that you have snippets that are the word ‚Äúlol‚Äù, followed by a 100-character long URL for an image. I'm not sure how this challenge should be handled.

    Exporting the data, APIs and UIs:

    Something I‚Äôve been very hesitant to add to the scope of this project is the application having integrations to allow it to automatically post a history if told to. This would be an incredibly useful function and wouldn‚Äôt be too hard to develop for a platform like slack as there are tools there that make it much easier like Block kit. But as my understanding of the challenges and skillset requirements is so limited, I‚Äôm not sure if delivering on this sort of functionality is possible. This is a ‚Äúnice to have‚Äù not part of the core set of features
